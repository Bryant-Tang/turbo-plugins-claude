# Plan

## 原則

- 本次工作處理 `<branch-name>` 分支上的需求，範圍以 [goal.md](../goal.md) 中**單一目標**（例如 `目標 2a`）為準；同數字的其他子目標（例如 `2b`、`2c`）由各自的 `goal-<id>/plan.md` 處理。
- 單一實作任務需控制在一個 chat session 可完成的範圍。
- 若整個目標的計畫怎麼切都還是塞不進一個 session，停下來回到 `write-goal` 將目標拆成更多同數字子目標（例如把 `2a` 拆成新的 `2a` + `2b`，原 `2b` 重編為 `2c`），不要硬寫超大的 `plan.md`。
- 每個實作任務都必須有 AC，完成後才能進入對應 review。
- 每個實作任務的 AC 都必須依固定 7 類分類撰寫，若不適用要明確寫 `N/A` 與原因。
- 每個實作任務都必須納入固定靜態檢查：整合性與相容性下的編譯風險、可維護性與程式碼品質下的 csharp-comment 規範、js-comment 規範與重用邏輯、可測試性與可觀測性下的診斷訊號。
- 最後一個實作任務固定為建置任務，不得與前面的功能任務合併。
- 任務拆分優先依功能邊界、風險邊界或可獨立驗證的邏輯邊界切開。
